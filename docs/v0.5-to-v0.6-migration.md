# Migrating from Vortex v0.5 to v0.6

v0.6 collapses the 36-atom v0.5 surface into 11 public tools — three task verbs
(`vortex_act` / `vortex_extract` / `vortex_observe`) plus eight kept atoms.
That trim drops the `tools/list` payload by ~69% and unlocks parameterised
tool merges (one `vortex_wait_for` instead of `vortex_wait` + `vortex_wait_idle`
+ `vortex_page_info`, etc.).

The biggest LLM-facing wins:

- **`tools/list` 14,500 B → 4,500 B** (~3,400 → ~1,000 tokens)
- **observe output** Markdown by default (-94% vs v0.5 JSON)
- **act effects** (`url_changed`, `ref_relocated`, `assigned_ref`, …) for
  chained reasoning without a follow-up `observe`
- **53 error codes** with prescriptive hints (next-action verb +
  tool/parameter suggestion)

Most caller code can be migrated mechanically. The rest of this doc tells you
how, and what falls outside the codemod.

---

## 1. Quickstart with `vortex-migrate`

`@bytenew/vortex-migrate` is a jscodeshift codemod CLI that rewrites v0.5 MCP
tool calls in your codebase to v0.6.

```bash
# install
npm install -g @bytenew/vortex-migrate@^0.6

# dry run (default) — print what would change
vortex-migrate ./src

# apply changes in place
vortex-migrate ./src --write

# JSON summary for CI / scripted workflows
vortex-migrate ./src --json
```

It recognises any `ObjectExpression` with `name` + `arguments` properties, so
it covers the typical SDK shape:

```ts
client.callTool({ name: "vortex_click", arguments: { target: "@e3" } });
```

…and any hand-rolled MCP request that follows the same payload shape.

### What you get out

- **Renames** (e.g. `vortex_click` → `vortex_act` with `action: "click"`
  injected) — applied silently when fully automatic.
- **Partial migrations** — applied with a warning when the v0.5 surface had
  knobs the v0.6 surface lacks (cookie-delete, url-based wait, console clear).
  Read the warning and verify the call.
- **Deletions** — `vortex_ping` is removed entirely (drops the surrounding
  `ExpressionStatement`).
- **Warn-only** — calls to tools that have no v0.6 equivalent are left
  untouched and a warning lists them so you can rewrite them by hand.
- **Indirect calls** — when the tool name is a variable (`{ name: tool, … }`),
  the codemod can’t prove what tool it is and warns instead of guessing.

The CLI exits 0 even when warnings are present — it’s informational, not a
gate. Use `--json` to feed warnings into your CI tooling.

## 2. The migration table

### 2.1 The 11 public v0.6 tools

| Tool | Purpose | v0.5 atoms it absorbs |
|---|---|---|
| `vortex_act` | Write actions on a UI element | `click`, `fill`, `type`, `select`, `scroll`, `hover` (+ `mouse_click` partial) |
| `vortex_extract` | Read visible text from page or element | `get_text` |
| `vortex_observe` | List interactive elements in scope | unchanged (also absorbs `frames_list` / `tab_list` semantically) |
| `vortex_navigate` | Navigate the active tab | unchanged |
| `vortex_tab_create` | Open a new tab | unchanged |
| `vortex_tab_close` | Close a tab | unchanged |
| `vortex_screenshot` | Capture page or element screenshot | unchanged |
| `vortex_wait_for` | Wait for element / idle / info | `wait`, `wait_idle`, `page_info` |
| `vortex_press` | Press a key globally | unchanged |
| `vortex_debug_read` | Read console / network logs | `console`, `network` |
| `vortex_storage` | localStorage / session / cookies CRUD | `storage_get`, `storage_set` |

### 2.2 Mechanical rewrites (16 atoms)

These are fully covered by `vortex-migrate`. Output column shows the v0.6
shape after the codemod runs.

| v0.5 call | v0.6 result | Notes |
|---|---|---|
| `vortex_click({ target })` | `vortex_act({ action: "click", target })` | |
| `vortex_type({ target, value })` | `vortex_act({ action: "type", target, value })` | |
| `vortex_fill({ target, value })` | `vortex_act({ action: "fill", target, value })` | |
| `vortex_select({ target, value })` | `vortex_act({ action: "select", target, value })` | |
| `vortex_hover({ target })` | `vortex_act({ action: "hover", target })` | |
| `vortex_mouse_click({ x, y })` | `vortex_act({ action: "click", x, y })` | **partial** — v0.6 act has no x/y; supply `target` ref |
| `vortex_get_text({ target })` | `vortex_extract({ include: ["text"], target })` | |
| `vortex_wait({ target, timeout })` | `vortex_wait_for({ mode: "element", value: target, timeout })` | **partial** — `url`-based waits need manual review |
| `vortex_wait_idle({ kind, idleMs })` | `vortex_wait_for({ mode: "idle", value: kind, timeout: idleMs })` | |
| `vortex_page_info({})` | `vortex_wait_for({ mode: "info" })` | |
| `vortex_console({ tail })` | `vortex_debug_read({ source: "console", tail })` | **partial** — `op: "clear"` has no v0.6 equivalent |
| `vortex_network({ tail, filter })` | `vortex_debug_read({ source: "network", tail, filter })` | **partial** — `op: "clear"` has no v0.6 equivalent |
| `vortex_storage_get({ scope, key })` | `vortex_storage({ op: "get"\|"session-get"\|"cookies-get", key })` | `op` derived from `scope` |
| `vortex_storage_set({ scope, key, value })` | `vortex_storage({ op: "set"\|"session-set", key, value })` | cookie-delete branch needs manual rewrite |

### 2.3 Deletion (1 atom)

| v0.5 | v0.6 |
|---|---|
| `vortex_ping` | **removed** — codemod drops the call site |

### 2.4 No automatic migration (15 atoms)

These tools have no v0.6 equivalent. The codemod leaves the call and emits a
warning so you can rewrite them by hand.

| v0.5 atom | Why no rewrite | What to do instead |
|---|---|---|
| `vortex_mouse_move` | Coordinate-based pointer move dropped | Express intent via `vortex_act(action="hover", target=ref)` |
| `vortex_mouse_drag` | Coords-based drag dropped | Tracked for v0.6.x; for now expand into multiple `vortex_act` calls |
| `vortex_get_html` | `extract` does not return HTML | Use `extract` text-only, or `screenshot` for visual |
| `vortex_evaluate` | Raw page-evaluate not exposed | Express intent via `act` / `extract`; rare cases need a Chrome DevTools session |
| `vortex_frames_list` | Frame enumeration not surfaced | Pass `frameId` per call; use `observe` per-frame |
| `vortex_tab_list` | Tab listing dropped | Track tabIds yourself across `tab_create` / `tab_close` |
| `vortex_history` | No `mode: "history"` in `wait_for` | Use `vortex_navigate` with `reload`/`url` |
| `vortex_network_response_body` | Bodies not exposed via `debug_read` | Capture with a CDP session; tracked for v0.6.x |
| `vortex_events` | Event stream removed | Switch to `console` / `network` sources |
| `vortex_storage_session` | No session import/export op | Persist via `cookies-get` + `session-set` manually |
| `vortex_file_upload` | No file_upload tool | Use `act(action="fill")` against `<input type="file">` |
| `vortex_file_download` | No file_download tool | Trigger via `act` + verify with `chrome.downloads` |
| `vortex_file_list_downloads` | No list_downloads tool | Query `chrome.downloads.search` directly |
| `vortex_batch` | No batch primitive | Sequence calls in your agent loop |
| `vortex_fill_form` | No fill_form helper | Expand to per-field `act(action="fill")` |

## 3. Other v0.6 surface changes

### 3.1 `target`: ref or descriptor

v0.5 took `selector` / `index` / `snapshotId` / `frameId` as separate fields.
v0.6 collapses them into a single `target` string:

| Form | Example | Meaning |
|---|---|---|
| Element ref | `"@e3"` | Element 3 in the latest snapshot |
| Cross-frame ref | `"@f1e2"` | Element 2 in frame 1 |
| CSS selector | `"button.primary"` | Selector against the active page |

Descriptors (role + name) arrive as the recommended form alongside the L3
reasoning resolver — see the L4 spec for the schema. The codemod does not try
to convert v0.5 selectors to descriptors automatically; existing selector
strings keep working.

### 3.2 `act` effects

`vortex_act` returns an `effects` object so the LLM can chain decisions
without a follow-up `observe`:

| Effect | When |
|---|---|
| `url_changed` | Action triggered navigation |
| `ref_relocated` | The ref now points to a different element (DOM moved) |
| `assigned_ref` | Descriptor input was resolved to a ref (returned for reuse) |
| `ref_state_change` | Element state flipped (e.g. `checked: false → true`) |
| `new_visible_elements` | New interactive elements appeared in viewport |

### 3.3 Error codes

53 codes total. Every error goes through `vtxError` and carries:

- `code` (machine-readable)
- `message` (50-300 chars, prescriptive)
- `hint.next_action` (verb-led: `call`, `verify`, `retry`, …)
- `hint.example` where useful

Invariants `I19` (no bare throws) and `I20` (no removed tool names in error
messages) gate this in the test suite.

## 4. LLM agent prompt updates

If you maintain agent system prompts, update:

- **Tool list section** — drop the 36 v0.5 tool names, list the 11 v0.6
  tools. The shorter `tools/list` should mostly do the work for you, but
  hand-written prompts often duplicate it.
- **Action chaining** — point the model at `effects.assigned_ref` and
  `effects.ref_relocated` as cheap signals to skip a fresh `observe`.
- **Error handling** — hints already tell the model the next action; explicit
  retry instructions in the prompt are usually now redundant.
- **Wait patterns** — `vortex_wait_for(mode="idle", value="dom")` replaces
  the older "click then poll" pattern in many flows.

## 5. Recommended migration workflow

1. **Branch** off main.
2. **Run `vortex-migrate ./src --json > migration-report.json`** to see the
   damage in machine-readable form.
3. **`vortex-migrate ./src --write`** to apply mechanical rewrites.
4. **Triage warnings**:
   - `<indirect>` warnings → replace the variable with a literal v0.6 tool
     name where possible.
   - `partial` warnings → check the partialNote and verify the rewritten
     args (cookie-delete, url-wait, console-clear, etc.).
   - warn-only entries → rewrite by hand using the table in §2.4.
5. **Run your test suite**. If you have an end-to-end agent harness, run a
   smoke task before merging.
6. **Update agent prompts** if you maintain any (see §4).
7. **Pin** `@bytenew/vortex-mcp@^0.6` in your dependency manifest.

## 6. v0.5 LTS

The `v0.5.x` maintenance branch is kept for at least two months after v0.6.0
ships. It receives critical bug fixes only — no new features, no atom
back-ports.

```bash
git checkout v0.5.x
```

If you cannot migrate now, pin to `v0.5.x` and migrate before the LTS window
closes.
